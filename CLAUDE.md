# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

This repo contains three subsystems for **The Talus Field**, a Yosemite editorial site and field guide product. They are intentionally co-located so a single change (e.g. brand rename, copy update) can touch all three at once.

| Path | What it is | Stack |
|---|---|---|
| repo root (`*.jsx`, `index.html`, `styles.css`, `bodies/`, `img/`) | **Editorial site** at `thetalusfieldjournal.com`: articles, interactive trip-planner map (`/map`), film archive (`/films`), packing checklists. Static, no bundler. | Vanilla React: `*.jsx` sources precompiled into `/dist/*.js` by `scripts/gen-compiled.mjs` (`index.html` loads the dist files); styled with hand-written CSS in `styles.css`. Edge SEO via the root Worker entry `edge/seo.js` (see `wrangler.jsonc`); response headers/CSP in `_headers`. |
| `apps/guide/` | **Field Guide PWA** at `talus-field-guide.pages.dev` (the custom domain `guide.thetalusfieldjournal.com` is deliberately unattached until launch). Account-gated **paid product** ($19 one-time, 18-month access; see "paid model" below), offline-capable. | Vite + React 19 + TypeScript + react-router-dom + zod. |
| `workers/` | **API** at `api.thetalusfieldjournal.com`. Auth, Stripe checkout/webhook, KV-backed buyer records, contact form, IndexNow, NPS programs feed (`/api/programs` + daily cron), NWS weather proxy (`/api/weather`), trip-link email (`/api/trip/email`), Google Calendar push (`/api/calendar`). | Cloudflare Worker + Hono + `@tsndr/cloudflare-worker-jwt` + KV. |

DEPLOY.md has a full deployment runbook (its "buy box / Field Guide launch checklist" is the current 8-step go-live sequence). Note that DEPLOY.md was written against the older trip-based PWA model (`1day`/`3day`/`5day`, "35 stops"). The PWA was since refactored to region-based (`valley`/`glacier-mariposa`/`tuolumne`/`hetch-hetchy`, ~34 stops); treat the smoke-test section as directionally correct but stale on specifics. `CONVERSION-STRATEGY.md` (July 2026) is the current reader→subscriber→buyer growth plan, successor to `GROWTH-AUDIT.md`.

## Common commands

Local dev hosts (matches `.claude/launch.json`):

```bash
# Editorial site — port 8765
python -m http.server 8765

# PWA — port 5173
npm --prefix apps/guide run dev

# Worker — local dev
cd workers && npm run dev      # = wrangler dev
```

Build / typecheck / checks:

```bash
npm --prefix apps/guide run build       # tsc -b && vite build → apps/guide/dist
npm --prefix apps/guide run lint
npm --prefix workers run typecheck      # tsc --noEmit

bash scripts/check-cache-busters.sh     # editorial: ?v= and BODY_VERSIONS sync
npm --prefix scripts run seo:check      # editorial: SEO mirror freshness
npm --prefix scripts run seo            # regenerate SEO mirrors
npm --prefix scripts run compile        # editorial: recompile *.jsx → /dist/*.js (run after any jsx edit)
npm --prefix scripts run prerender      # editorial: regenerate /prerender article fragments
npm --prefix scripts run check          # editorial: all pre-commit guards (cache-busters + seo + prerender + compile checks)
npm --prefix scripts run images         # regenerate responsive image variants (sharp)

npm --prefix scripts run checks         # editorial: offline system-checks suite (SEO/render/distribution)
npm --prefix scripts run checks:online  # + link liveness, sitemap 200s, template smoke, SPF/DKIM/DMARC
```

Dev tooling lives in `scripts/` with its own `package.json`, deliberately not at the repo root, so the root stays a static, build-free deploy. Run `cd scripts && npm install` once before using the npm scripts above. The root `package.json` is NOT a tooling home: it exists only because the Cloudflare Workers Build for the editorial worker runs `npm run build` before deploying, and its `build` script is a no-op echo. Do not add dependencies or real build steps to it (`.assetsignore` also excludes it from the asset upload).

Deployment is **not** done from this repo's CI. On merge to `main`: the **editorial site** deploys via a git-connected Cloudflare **Workers Build** (root `wrangler.jsonc`; it runs the root `npm run build` shim then `npx wrangler deploy`, uploading the repo root as static assets with `edge/seo.js` as the Worker entry); the **PWA** deploys via the git-connected Cloudflare **Pages** project `talus-field-guide` (root dir `apps/guide`, build `npm run build`, output `dist`, Node pinned by `apps/guide/.nvmrc`). The **API Worker** never auto-deploys: it requires a manual `wrangler deploy` from `workers/` (with KV namespace IDs filled into `wrangler.toml` and secrets set via `wrangler secret put`).

## Editorial site conventions

- **No bundler, but a compile step.** The `*.jsx` files at the root are the source of truth, but the browser loads the precompiled `/dist/*.js` versions (no in-browser Babel). After editing any root or `bodies/` jsx, run `npm --prefix scripts run compile` and commit the regenerated `dist/` files alongside the source; `npm --prefix scripts run check` fails if they drift. Edge SEO head rewriting is done by `edge/seo.js`, the root Worker's fetch handler (ported from the retired Pages Function `functions/_middleware.js`); it also injects crawler prose for hub routes (`HUB_PROSE`) and serves unknown paths as real 404s.
- **Route-level lazy loading (July 2026 LCP pass).** `index.html` eagerly loads only the shared shell: the self-hosted React UMD files under `/vendor`, `storage.js`, `affiliate.js`, `data.js`, `dist/tweaks-panel.js`, `dist/components.js`, `dist/app.js`. Every `page-*.js` bundle (plus `videos-data.js`, `itineraries-data.js`) loads on demand via `PAGE_MODULES`/`ensureRoute` in `app.jsx`: the active route's bundle before first render, the rest prefetched on first user interaction. **Adding a page means registering it in `PAGE_MODULES` (app.jsx), not adding a `<script>` tag to index.html.** Injected scripts reuse the shared `?v=` (read from the app.js tag), so the cache-buster discipline is unchanged. Only two Google Font families load (EB Garamond + Inter); don't reintroduce the others.
- **"This Week in the Park" (`/now`)** is the weekly dispatch page: entries live in `now.json` (hand-edited weekly, newest first; see its `__comment` for the workflow), rendered by `page-now.jsx` (own `?v=` counter on `NOW_URL`, like the map's `POINTS_URL`), served to non-JS crawlers via the `HUB_PROSE` entry in `edge/seo.js`, and its sitemap lastmod tracks the newest dispatch automatically. It is a standing editorial commitment: a stale dispatch is worse than none.
- **Cache-buster discipline.** When you edit a JSX file or `styles.css`, bump its `?v=N` query in `index.html`. Cloudflare and browsers both cache aggressively otherwise. All page scripts and the stylesheet share one version number; bump them together to a new shared number when shipping a batch. `points.geojson` has its **own** `?v=` counter: the `POINTS_URL` const in `page-map.jsx` (exported as `window.POINTS_URL`, also consumed by `/itineraries`) — bump it separately when editing pins. Run `bash scripts/check-cache-busters.sh` before committing — it errors if any reference is missing a `?v=` or if `window.BODY_VERSIONS` is out of sync with `bodies/`.
- **Globals via `window`.** Each JSX file attaches its top-level component to `window` (e.g. `window.GuidePage = GuidePage`) so siblings can reference it. The list of `/* global */` comments at the top of each file declares what it consumes. The full `window.*` API surface, the GA4 event inventory, and the localStorage key inventory are documented in `ARCHITECTURE.md`. All localStorage access goes through `window.safeStorage` (storage.js), never `window.localStorage` directly.
- **Article bodies** live in `bodies/*.jsx` and are **lazy-loaded**, not registered as `<script>` tags in `index.html`. `data.js` indexes each article in `window.ARTICLES` and versions each body per-slug in `window.BODY_VERSIONS`; `window.loadArticleBody(slug)` injects the script on demand. When you add or edit a body, add/bump its `BODY_VERSIONS` entry.
- **SEO mirrors are generated, not hand-edited.** `data.js` (`window.ARTICLES`) is the single source of truth for the catalog; SEO enrichment the browser does not need (keywords, wordCount, FAQ, trail facts) lives in `seo-data.json` keyed by slug. `articles.json` (read by the edge SEO function `functions/_middleware.js`), `sitemap.xml`, `feed.xml`, and the article list in `llms.txt` are all generated by `scripts/gen-seo-artifacts.mjs`. After adding or editing an article (data.js + its body) and any enrichment (seo-data.json), run `npm --prefix scripts run seo`, then commit the regenerated files. Run `npm --prefix scripts run seo:check` (or `bash scripts/check-seo-artifacts.sh`) before committing — it fails if any mirror is stale. FAQ answers and trail numbers in `seo-data.json` must come from the article's published body; do not invent facts.
- **System checks suite.** `scripts/system-checks.mjs` (run via `npm --prefix scripts run checks`) is a health battery with one module per check in `scripts/checks/` (links, per-page SEO + og:image + Article JSON-LD, sitemap, feed, image hygiene, mirror freshness, template smoke, email-auth). It exits non-zero only on **error**-severity findings; length/size/keyword issues are **warnings**. Offline by default; `--online` adds link liveness, sitemap 200s, the template smoke test, and SPF/DKIM/DMARC DNS lookups. The catalog loaders are shared with `gen-seo-artifacts.mjs` via `scripts/lib/catalog.mjs` (one `node:vm` harvest of `data.js`). It runs nightly in CI (`.github/workflows/system-checks.yml`), which also records a Lighthouse Core Web Vitals trend to `scripts/data/lighthouse-history.json` (warn-only on regression). New-URL search-engine notification is **not** here — `indexnow.yml` already push-indexes on publish, and standalone Google/Bing sitemap pinging was retired in 2023. The committed baseline is `scripts/data/baseline-report.md`.
- **Map page** (`page-map.jsx`, `/map` route): Google Maps JS API trip builder over `points.geojson`. The API key sits in `index.html`, domain-restricted in the Google Cloud console; `mapId` is required by AdvancedMarkerElement. Pins are maintained **by hand** in `points.geojson` (it was seeded once from the PWA's stops via `scripts/seed-points-from-stops.mjs`; re-running overwrites the file). The trip persists in localStorage (`tfg.trip`) and is shareable via `/map?trip=id1,id2`. The map is browsable by everyone; the trip *builder* is newsletter-gated (`tfg.map.unlocked`), bypassable by design and fails open when storage is unavailable. A July 2026 UX pass added category filter chips (`?cat=` param, `CATEGORY_STYLES`), stop search, marker clustering (MarkerClusterer lazily injected from unpkg below zoom 13, fails soft to plain markers; unpkg is CSP-allow-listed), a peek/half/full mobile bottom sheet, and undoable trip-change toasts. Quick-pick itineraries are derived from `window.ITINERARIES`, and a "Email this trip to yourself" box (`TripEmailBox`) submits to Buttondown **and** POSTs `{email, stops, website}` to the Worker's `/api/trip/email`, which builds the share URL server-side and mails it via Resend.
- **Itineraries + Conditions pages** (July 2026). `itineraries-data.js` is a raw (uncompiled) script like `data.js`: it defines `window.ITINERARIES` (four plans: `1day`/`2day`/`3day`/`halfday`, each day listing `points.geojson` ids in drive order) and `window.getItineraryStopIds(id)`; it is the single source of truth for both `/itineraries` (`page-itineraries.jsx`, which fetches `window.POINTS_URL` to resolve stop names and links each plan to a real `/map?trip=…` navigation) and the map's quick picks. `edge/seo.js` carries a **hand-maintained** TouristTrip ItemList for `/itineraries` — keep it in sync when plans change. `/conditions` (`page-conditions.jsx`) is a bookmarkable link-out page (webcams, entrance waits, hardcoded NWS point-forecast links) reusing `WebcamStrip` and `EntranceWaits` from components.jsx; it fetches nothing itself. Adding a static route like these touches: `app.jsx` (route + `REQUIRED_GLOBALS`), `index.html` script tag, the compile list in `scripts/gen-compiled.mjs`, `STATIC_ROUTES` in `scripts/lib/catalog.mjs`, an `edge/seo.js` entry, then `npm --prefix scripts run seo`.
- **Film archive** (`page-films.jsx` + `videos-data.js`, `/films` route): the NPS *Yosemite Nature Notes* series as `window.NATURE_NOTES`. Embeds are click-to-load facades — only the YouTube thumbnail loads until the reader presses play, then a `youtube-nocookie.com` iframe replaces it. `youtubeId` values were verified against the published uploads; do not edit one without checking the video it points to. Deks are original Talus Field copy, not NPS copy.
- **Responsive images.** `scripts/gen-responsive-images.mjs` writes AVIF/WebP/JPEG variants at several widths into sibling `responsive/` folders (under `img/` and `apps/guide/public/photos/`), slug-named so the in-browser `ResponsiveImage` helper derives URLs with no manifest. Run `npm --prefix scripts run images` after adding source JPEGs.
- **CSP lives in `_headers`.** Any new third-party script, frame, or connect target must be allow-listed in the Content-Security-Policy there or it will be blocked in production (it works locally, which is the trap).
- **`page-guide.jsx` runtime URL overrides:** `window.GUIDE_API_BASE` and `window.GUIDE_APP_BASE` can be set in a console snippet to point at local dev (`http://localhost:8787` and `http://localhost:5173`) before the script loads.

## PWA architecture

- **Content model.** `apps/guide/src/content/schema.ts` defines `Stop` (zod schema) with a `region` enum: `valley` | `glacier-mariposa` | `tuolumne` | `hetch-hetchy`. `content/stops.ts` is the seed array, validated at module load via `Stops.parse(seed)` — schema violations throw in the Vite overlay or fail the build. `content/index.ts` exports `REGIONS` (the picker metadata), `REGION_SHORT` (chip-length labels), and helpers `getStopsByRegion`, `getStopById`, `getRegionMeta`. `content/itineraries.ts` defines 1/2/3-day presets whose day stop lists are derived live from `getStopsByRegion()` — adding a stop to a region flows through with no itinerary edit (the presets cover the original three regions; Hetch Hetchy is a la carte). `content/seasonal.ts` is the bundled seasonal almanac (full moons, road-opening windows, firefall): entries convert to ProgramEvent records (source `seasonal`, an enum value mirrored into the Worker schema) and merge into /programs client-side, so the agenda keeps an offline floor; `typical`-confidence entries are labeled as such everywhere they render.
- **Coords are web-verified, not ground-truthed.** A July 2026 pass checked every `coord` in `stops.ts` against NPS place/trailhead pages, USGS GNIS, and OSM-derived sources; each coord line carries its source and any prior offset. Stops still marked `// TODO: verify on the ground` (unsigned pullouts, off-trail features, conflicting sources) have no authoritative source; never strip those markers without a real field visit, because the field guide's value prop depends on accurate parking turnouts.
- **Routes** (see `App.tsx`): `/open` (magic-link landing), `/login`, `/` (region picker with per-region five-day forecasts), `/welcome` (first-run orientation: offline explainer, install step, download packs; `/open` and `/` route here until `tfg.onboarded` is set), `/region/:regionId`, `/stop/:stopId`, `/essentials` + `/essentials/:topicId` (know-before-you-go, incl. the `before-you-go` night-before checklist and the packing checklist), `/secret-guide` (the merged premium section: region-less secret spots + hidden-collection stops, filtered by `SecretCategory` tabs; `/secret-spots` and `/hidden-areas` redirect here), `/search`, `/programs` (trip-dates → date-grouped program list), `/trip` (day-by-day planner; the review step's calendar sheet offers a hosted subscription feed or a one-time ICS), `/map` (MapLibre GL over the Worker's `/tiles` raster topo proxy — NOT Google Maps; pin styles and the Google Maps directions deeplink live in `src/map/kinds.ts`), `/account` (DownloadManager). Everything except `/open` and `/login` is wrapped in `RequireAuth`. `/trip/*` subpaths redirect to `/trip`.
- **Onboarding** (`lib/onboarding.ts`, `routes/Welcome.tsx`): flag `tfg.onboarded`, deliberately **fails open** — a storage read error counts as onboarded so storage-denied browsers aren't trapped on `/welcome`. Only the front page redirects there; deep links are never intercepted.
- **PWA map** (`routes/Map.tsx`): one MapLibre instance created once and never unmounted; URL carries `?tab=&itinerary=&stop=&kinds=&secret=`. Pin-kind filter chips with live counts (`null` set = "All"; validate `?kinds=` with `Object.hasOwn`, not `in`), a Secret Guide toggle, browse-by-area grouping (REGIONS + a region-less "Secret spots" group; docks as a bottom handle on phones), a "Near you" list of the five closest *visible* stops (`utils/geo.ts` haversine), and map-only amenity pins (parking/campgrounds from `AMENITIES`, separate marker pipeline, no trip actions).
- **Weather feature** (`src/weather/`). Same shape as programs: `schema.ts` mirrors `workers/src/lib/weather.ts` by hand, `useWeather` is network-first against `/api/weather` with a Cache API fallback (payload in `tfg-data` under a synthetic key, stamp in `tfg.weather.meta`). Renders as `RegionForecast` (five-day hi/lo inside each Home region card) and `WeatherStrip` (next four periods on region pages) — not on stop pages. Weather is garnish, never an error: stale > 12 h gets a "forecast is old" label, stale > 48 h renders nothing (`staleness.ts`). `forecastDays.ts` folds NWS day/night half-periods into park-local calendar days.
- **Auth state.** JWT lives in `localStorage` under key `tfg.jwt` (`auth/storage.ts`). Session shape: `{ jwt, username }` — the JWT `sub` claim is treated as a string label and surfaced as `username` regardless of which Worker auth path issued it. The session is derived from the JWT itself (`sessionFromJwt`), not by re-reading localStorage, so sign-in works in storage-blocked browsers without burning single-use magic-link tokens.
- **PWA localStorage keys** beyond `tfg.jwt`: `tfg.accessEndedAt`, `tfg.me`, `tfg.downloads` (pack completion), `tfg.favorites`, `tfg.checklist`, `tfg.install.dismissed`, `tfg.beforeYouGo.dismissed`, `tfg.onboarded`, `tfg.trip.dates` (shared by `/programs` and `/trip`), `tfg.trip.plan` (the planner, zod-parsed drop-on-corrupt), `tfg.trip.feed` (hosted calendar-feed record: token + URL + last-publish stamp), `tfg.calendar.google`, `tfg.programs.meta` and `tfg.weather.meta` (cache stamps). Cache API buckets: `tfg-data` (programs + weather payloads, unversioned, survives deploys), `tfg-tiles` (map tiles), `tfg-runtime` (photos/fonts/icons), `tfg-shell-<version>` (app shell, rotated per deploy).
- **API calls** go through `apiFetch` (`lib/api.ts`), which applies a 15-second `AbortSignal.timeout` by default (captive-portal Wi-Fi accepts connections and never answers); a timeout rethrows as a plain network `Error`, not `ApiError`. Callers passing their own `signal` keep control.
- **Programs feature.** `src/programs/schema.ts` mirrors `workers/src/lib/programs.ts` (keep in sync by hand — no shared package on purpose). `usePrograms` is network-first against `/api/programs` with an offline fallback to the cached window. The trip planner (`src/trip/`) snapshots program events into the plan so it survives cache eviction; `trip/ics.ts` hand-rolls the ICS (Pacific VTIMEZONE, folded lines, stable UIDs, VALARM reminders on timed events, calendar-level X-WR-CALNAME/refresh hints) and `trip/exportTrip.ts` prefers Web Share with a File (iOS standalone PWA can't reliably download blobs) before falling back to an anchor download.
- **Calendar integration.** Two paths from the trip review's "Add to calendar" sheet (`components/TripCalendarSheet.tsx`): a one-time `.ics` file (fully offline), and a hosted subscription feed. Subscribing renders the ICS client-side and POSTs it to the Worker (`trip/feed.ts`); the sheet then offers webcal (Apple) and Google Calendar add-by-URL links. `trip/feedSync.ts` (started in `main.tsx`) republishes automatically after plan edits, debounced, while online; failures are silent and surface only as a stale "updated" stamp. Slotting travel buffers are distance-derived (`trip/slotting.ts`: haversine at a 22 mph park heuristic + park-and-walk, clamp 10-75 min, flat 30 when a coord is missing).
- **Service worker** (`public/sw.js`) caches the app shell + photos for offline use. The `UpdateBanner` component prompts when a new build is available. **Hard invariant since the July 2026 audit:** an HTML response (the SPA `_redirects` 200 fallback) must never be cached under an asset/photo/tile/shell key — the SW and `useDownloads.fetchIntoCache` both guard with an `isHtml` check and treat such a response as a failed URL, and `activate` self-heals by purging HTML from the runtime and tile caches. Photo pre-warming goes through `pwa/precache.ts` (`PRECACHE_URLS` message, prod-only, works pre-controller via `serviceWorker.ready` with a 10 s timeout). The `beforeinstallprompt` event is captured at module level in `pwa/installPrompt.ts`, called from `main.tsx` **before render** (component listeners fire too late); `Welcome.tsx` and `InstallPrompt.tsx` consume it via `useDeferredInstallPrompt`. `vite.config.ts` pins React and the router into a `react-vendor` manual chunk so it survives shell re-precache across deploys.
- **Download packs** (`offline/useDownloads.ts`): live status and abort controllers are module state so in-flight downloads survive remounts; `verifyPack` tolerates a small missing fraction (5% for the tile pack) and records tolerated failures rather than demanding 100%.
- **Stop photos are a placeholder pass.** `public/photos/` reuses editorial-site images (several stops share the same photo); many stops still have no matching image and need dedicated photography before launch.

## Worker architecture

Routes mounted in `workers/src/index.ts`:

- `/api/auth/login` — **legacy** email + 6-digit code path, KV-backed buyer lookup. Currently unused by the PWA but kept for when Stripe checkout comes back.
- `/api/auth/dev-login` — **active** username + code path, env-backed (`DEV_USERNAME`/`DEV_CODE`/`ADMIN_USERNAME`/`ADMIN_CODE`). Pre-Stripe; either pair issues a JWT.
- `/api/auth/exchange` — exchanges a magic-link `accessToken` for a JWT.
- `/api/checkout/*` — Stripe Checkout session start + inventory cap enforcement. The editorial `/guide` page still calls `/api/inventory` for the scarcity counter.
- `/api/stripe/webhook` — Stripe webhook handler for `checkout.session.completed` (provisions a buyer record in KV — `expiresAt` = purchase + 548 days, a true 18 calendar months — and emails the access code) and `charge.refunded` (any refund, full or partial, revokes access by setting `expiresAt` and `refundedAt` to now; missing email or unknown buyer is logged for manual revoke, not failed). Idempotent via a `stripe_event:<id>` dedupe key. Both events must be enabled on the Stripe webhook endpoint — see the DEPLOY.md launch checklist.
- `/api/contact` — editorial contact form backend (`page-newsletter-contact.jsx` posts here). Validates a fixed subject enum, uses a `website` honeypot field against bots, and relays via Resend (`lib/email.ts`).
- `/api/indexnow/submit` — bearer-token-gated POST that forwards a `{ urls: [...] }` list to api.indexnow.org. Push-indexes Bing / Yandex / Seznam / Naver / Yep in a single call. Use `scripts/indexnow-ping.sh` after publishing an article. Requires `INDEXNOW_KEY` and `INDEXNOW_ADMIN_TOKEN` secrets.
- `/api/programs?start=&end=` — **unauthenticated** merged park-program listings for a trip window (≤31 days): NPS Events API data from the `GUIDE_PROGRAMS` KV cache plus the hand-curated `src/data/manual-programs.ts` (Conservancy / Aramark / astronomy — the NPS API doesn't carry most partner events). A daily cron (`[triggers]` in wrangler.toml, `scheduled` export in `index.ts`) ingests a rolling 120-day window; the route self-heals with a live fetch + KV backfill when the cache is missing or >48h old. Needs the `NPS_API_KEY` secret (free, developer.nps.gov); without it the feed serves manual curation only. Seasonal curation workflow: edit `manual-programs.ts` (zod-validated at load), `wrangler deploy`.
- `/api/trip/feed` — the PWA's hosted calendar feed (`routes/trip.ts`). POST (JWT) stores the client-rendered ICS in `GUIDE_BUYERS` KV under a per-account capability token (`tripfeed:<token>` / `tripfeedToken:<sub>`, 400-day TTL refreshed per publish, 30 writes/hour/sub); GET `/feed/<token>.ics` serves it **unauthenticated** (the 64-hex token is the auth — calendar pollers can't send headers) with revocation via the buyer record (expired/refunded → 410, missing record = operator → allow); DELETE revokes both keys. CORS `allowMethods` includes DELETE for this route.
- `/api/trip/email` — **unauthenticated** editorial trip-link mailer (`routes/trip-email.ts`; mounted before `/api/trip` on purpose). POST `{email, stops, website}` from the editorial map's TripEmailBox; `website` is a honeypot (non-empty → fake 200). Validates and dedupes stop ids (≤30, `[a-z0-9-]`), builds the `/map?trip=…` URL server-side (so it can't mail arbitrary links), sends via Resend (`sendTripLink` in `lib/email.ts`), and rate-limits to 5 sends/hour per SHA-256-hashed IP in `GUIDE_BUYERS` KV.
- `/api/calendar/google/*` — Google Calendar push for the PWA trip planner (`routes/calendar.ts`). Server-side OAuth code flow; the refresh token never leaves the Worker (the client only learns "connected" + the email). The plan is reconciled into the buyer's primary calendar and re-synced after edits. JWT-gated writes, KV keyed by sub, hourly write counter — same shape as `routes/trip.ts` except the OAuth callback, which is a top-level redirect authenticated by a single-use `state` stored in KV.
- `/api/weather` — **unauthenticated** NWS forecast proxy (`routes/weather.ts`, `lib/weather.ts`). No params; returns all four fixed park spots (ids match the PWA Region enum). Two-step api.weather.gov fetch (points → gridpoint forecast), keyless (User-Agent required, no secret). Cached in `GUIDE_PROGRAMS` KV (`weather:v1`, plus a gridpoint-URL cache); the route refreshes inline when the record is missing or >2 h old and a failed spot keeps its stale copy; responses carry `max-age=900`. The daily cron also calls `refreshWeather` as a floor. The zod schema is hand-mirrored in `apps/guide/src/weather/schema.ts`.
- `/api/inventory` also returns `priceCents` (from `GUIDE_PRICE_CENTS` in wrangler.toml) — the editorial buy box renders it live, so the price is edited in exactly one place.

Both auth paths sign the same JWT shape (`{ sub, iat, exp }`, HS256, 90-day TTL) using `MAGIC_LINK_SIGNING_SECRET`. The `sub` claim carries an email for `/login` and a username for `/dev-login` — the PWA doesn't care which.

CORS in `index.ts` allow-lists the editorial origin, the PWA origin, and `http://localhost:5173`/`5174`/`8000` for local dev. Add new dev origins there.

## Brand & voice

The brand is **The Talus Field** (formerly "Yosemite Sentinel"). The editorial voice is dry, declarative, journalistic — no marketing fluff, no exclamation marks. House style replaces em-dashes in reader-facing copy with commas, colons, or periods; a deliberate sweep removed them from article bodies, so don't reintroduce them when writing copy. Match the existing tone.

## Things that have surprised past edits

- The editorial site has **no test suite, lint, or typecheck** — runtime errors only surface in the browser. The pre-commit guards are `npm --prefix scripts run check` (cache-busters + SEO mirrors + prerender + compiled-dist freshness in one command) plus the offline battery `npm --prefix scripts run checks`.
- `DEPLOY.md` references `1day`/`3day`/`5day` trips and "35 stops"; the current model is region-based with ~34 stops across four regions.
- **The guide is a paid product ($19 one-time, 18-month access) currently in waitlist mode.** `page-guide.jsx` has a `GUIDE_ON_SALE = false` flag: it renders `GuideWaitlistBox` (a Buttondown capture tagged `guide-waitlist`) instead of the intact `GuideBuyBox` (which POSTs `/api/checkout/start` and reads the price from `/api/inventory`; `GUIDE_PRICE_CENTS = "1900"` in `workers/wrangler.toml` is the single source of truth). Every spot that changes at launch is marked with a `GUIDE-LAUNCH` comment (`page-guide.jsx`, `app.jsx`, `edge/seo.js`, `index.html`, and the `GUIDE_LISTED` flag in `scripts/gen-seo-artifacts.mjs`) — flipping to on-sale means visiting every marker. Going live also requires ops steps outside the tree — KV namespace ids, secrets, the Stripe webhook (both events), Resend domain verification — see the launch checklist in DEPLOY.md. `/api/auth/dev-login` remains the operator door; rotate `DEV_*` creds before launch.
- The masthead issue label (`window.SITE.issue` / `issueDetail` in `data.js`) derives its month from the current date at runtime — don't hard-code a month name into masthead copy.
- `points.geojson` carries its own `?v=` cache-buster (`POINTS_URL` in `page-map.jsx`), separate from the shared `?v=N` that all page scripts use; editing pins without bumping it ships stale map data.
- The PWA service worker will happily serve a cached SPA-fallback HTML page as a photo, tile, or the shell if the `isHtml` guards are removed — this bricked offline launch once. Treat "never cache HTML under a non-navigation key" as an invariant when touching `sw.js` or `useDownloads.ts`.
- PWA stop photos look wired up but are placeholders recycled from the editorial site; several stops share the same image and the comment block at the top of `stops.ts` documents this.
